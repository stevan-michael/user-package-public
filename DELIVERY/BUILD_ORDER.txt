======================================================================
1. FOUNDATION
======================================================================

Purpose:
Define the authoritative responsibility, interpretation,
applicability, and operational boundary of Build Order.

Build Order Contract Interpretation:
Every implementation requirement expressed using "shall"
constitutes a mandatory implementation constraint.

Implementation behavior not explicitly required or permitted by
this Standard or a higher-authority USER_PACKAGE document shall
not be assumed to be permitted.

Requirements owned by another authoritative USER_PACKAGE document
shall not be interpreted as implementation requirements.

Build Order Definition:
A Build Order is the approved implementation package that
translates locked design and project knowledge into authorized
implementation requirements.

Applicability:
This Standard applies only to approved implementation activities.

Excluded Activities:
- Documentation authoring.
- Documentation review.
- Architecture design.
- Governance.
- Standards authoring.
- Runtime orchestration.
- Output rendering.

Responsibility Rule:
Build Order governs implementation authorization only.

======================================================================
2. CORE PRINCIPLES
======================================================================

1.

Build Order represents the final
implementation state.

Discussion, research, reviews, and alternatives
are complete before a Build Order is created.

2.

Build Order is implementation-focused.

It explains what shall be implemented.

It does not explain project history.

3.

Build Order is self-contained.

A compatible Implementation Assistant shall be
able to implement the project using only:
- Runtime.
- Approved Standards.
- Current Project Documentation.
- Build Order.

Previous conversation history shall not be required.

4.

Build Order never replaces Current Project
Documentation.

It translates approved design into authorized
implementation requirements.

5.

Build Order remains concise.

Only implementation-relevant information
belongs inside the document.

Historical discussion, rejected ideas, and design
exploration belong elsewhere.

6.

Build Order assumes all architectural decisions have
already been approved.

The Implementation Assistant executes approved decisions.

The Implementation Assistant does not redesign architecture.

======================================================================
3. CREATION CONDITIONS
======================================================================

Purpose:
Define when a Build Order shall and shall not be created.

When To Create:

Create a Build Order only when implementation is ready.

Typical workflow:

Discussion
↓
Research
↓
Review
↓
Architecture Lock
↓
Approval
↓
Build Order
↓
Implementation
↓
Verification

When Not To Create:

Do not create a Build Order when:
- Requirements are still changing.
- Architecture remains under discussion.
- Major blockers remain.
- Project direction is uncertain.
- Multiple implementation candidates still exist.

======================================================================
15. DELIVERY ENFORCEMENT
======================================================================

The selected Delivery Mode is mandatory.

The Implementation Assistant shall deliver the completed
implementation using the selected Delivery Mode.

The Implementation Assistant shall not replace the selected
Delivery Mode.

When Script Delivery is selected, deliver one
complete ready-to-run Python writer script.

Do not substitute:
- PATCH_DELIVERY_STANDARD.
- Partial Patch Delivery.
- Consecutive payloads.
- Multi-message delivery.
- Incremental delivery.
- Manual reconstruction.

A platform limitation does not authorize changing the Delivery Mode.

If the selected Delivery Mode cannot be completed safely,

stop and report the blocker.

Only the user may authorize another Delivery Mode.

Delivery Ownership:
The selected Delivery Mode defines delivery behavior and shall
not be redefined by Build Order.

======================================================================
17. VALIDATION
======================================================================

Before delivery verify:
- Every required file has been generated.
- Folder structure is complete.
- Syntax has been verified.
- Internal consistency has been verified.
- No placeholder content remains.
- No incomplete sections remain.
- Portable requirements are satisfied.
- Documentation is included.
- Ready to run.
- Implementation Rules are respected.
- Applicable Standards have been identified.
- Standards Compliance Review has completed.
- Identified compliance issues have been corrected.
- Delivery Mode has been respected.
- Current Project Documentation has been
  preserved.
- No unauthorized architectural changes
  have been introduced.

Validation Result:
- PASS: Every applicable implementation requirement is satisfied.
- FAIL: One or more implementation requirements are violated.
- NON-APPLICABLE: A requirement does not apply.

======================================================================
18. IMPLEMENTATION ASSISTANT RESPONSIBILITIES
======================================================================

The Implementation Assistant shall:
- Implement every approved requirement.
- Follow Runtime.
- Follow approved Standards.
- Preserve project architecture.

Implementation Assistant Ownership Interpretation:

Executing an approved Build Order does not
transfer ownership of the project,
architecture, requirements, or Standards to
the Implementation Assistant.

Authority remains defined by the applicable
authoritative USER_PACKAGE documents.

Implementation Preservation Interpretation:

Preserving approved architecture, approved
requirements, and approved Standards takes
precedence over implementation optimization.

Equivalent implementation requirements shall
produce equivalent implementations unless
explicitly instructed otherwise or required by
an approved architectural revision.

- Avoid redesign.
- Avoid optimization outside the approved scope.
- Avoid introducing new concepts.
- Report blockers instead of inventing solutions.
- Preserve compatibility with the Current Project
  Documentation.
- Report conflicts instead of silently resolving
  them.

======================================================================
19. AI EXECUTION RULES
======================================================================

When a Build Order is supplied,

treat it as the highest implementation authority
for the current task.

Assume every approved decision is final.

Do not reinterpret requirements.

Do not redesign architecture.

Do not introduce additional features.

Do not remove approved features.

Do not optimize beyond the approved implementation.

If ambiguity exists,

use the Current Project Documentation.

If ambiguity remains,

stop and report the ambiguity.

Never invent missing requirements.

When implementation requirements conflict with
previous conversation history,

Build Order takes precedence unless explicitly
superseded by the user.

Implementation Decision Rule:

Implementation decisions shall be derived only
from:
- This Build Order.
- Applicable higher-authority USER_PACKAGE
  documents.
- Current Project Documentation.
- Explicit User instructions.

Implementation decisions shall not be derived
from Implementation Assistant preference or implementation style.

20. METHOD PRECEDENCE
======================================================================

When a Build Order references an active Method,

execute according to that Method.

The Method defines:
- Transformation
  behavior.

The Delivery Mode defines:
- Delivery behavior.

Do not reinterpret, replace, or bypass the Method.

If:
- Build Order,
- Method,
- and Current Project Documentation

are consistent, begin implementation immediately.

Only a verified blocker may suspend execution.

Report blockers instead of changing the Method or
Delivery Mode.

======================================================================
25. BUILD ORDER PHILOSOPHY
======================================================================

Build Order separates design from implementation.

Architects design.

Documentation preserves.

Build Order authorizes implementation.

It does not authorize redesign.

Implementation Assistants execute.

This separation enables consistent implementation
across different assistants, AI models,
conversations, delivery sessions,
operating environments, and future projects while
preserving approved architectural decisions.

======================================================================
26. REVISION POLICY
======================================================================

Purpose:
Define when Build Order Standard may be revised.

Revision Criteria:
This Standard may be revised only when validated implementation
work demonstrates:

- Authorization ambiguity.
- Repeated implementation interpretation failure.
- Missing implementation coverage.
- Delivery enforcement failure.
- Standards compliance failure.
- Responsibility overlap.
- Compatibility failure.
- Conflict with a higher-authority document.
- Proven maintenance cost.

Revision Rules:

Revision shall:
- Preserve the primary responsibility.
- Preserve approved implementation authority.
- Preserve compatibility when requirements remain valid.
- Resolve validated deficiencies before adding new fields.
- Preserve Single Authority Ownership.

Revision shall not:
- Absorb architecture design.
- Absorb Runtime behavior.
- Absorb Delivery Mode behavior.
- Absorb documentation responsibilities.
- Introduce hidden assumptions.
- Authorize redesign without explicit User approval.
