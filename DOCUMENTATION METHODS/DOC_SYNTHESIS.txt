======================================================================
1. FOUNDATION
======================================================================

Purpose:
Define the authoritative responsibility, interpretation,
applicability, and operational boundary of File Synthesis.

Synthesis Contract Interpretation:
Every synthesis requirement expressed using "shall" constitutes a
mandatory documentation transformation constraint.

Synthesis behavior not explicitly required or permitted by this
Method or a higher-authority USER_PACKAGE document shall not be
assumed to be permitted.

Requirements owned by another authoritative USER_PACKAGE Method
or Standard shall not be interpreted as File Synthesis
requirements.

Synthesis Definition:
File Synthesis generates one new document by applying approved
Source information to the approved Baseline presentation model.

Authority Model:
Source documents define content. The Baseline defines
presentation. This Method defines transformation.

Applicability:
This Method applies only to documentation assets within its
defined Scope.

Excluded Assets:
- Source code.
- Runtime implementations.
- Executable assets.
- Behavioral Standards.
- Binary assets.
- Machine-readable serialization formats.

Responsibility Rule:
File Synthesis governs document synthesis only.

======================================================================
2. INPUT CONTRACT
======================================================================

Baseline:
The authoritative structural template.

Preserve:
- Document structure.
- Section hierarchy.
- Writing style.
- Grammar style.
- Property format.
- Naming conventions.
- Formatting.
- Visual layout.
- Overall document philosophy.

Synthesis Applicability:

This Method applies only to documentation assets
within its defined Scope.

This Method shall not be applied to:
- Source code.
- Runtime implementations.
- Executable assets.
- Behavioral Standards.
- Output rendering.

Assets outside the defined Scope remain governed
by their applicable authoritative Method or
Standard.

Source:
The authoritative information source.

Supported Input:
- One Source document.
- Multiple Source documents.

Supported document types depend on the
assistant's available capabilities.

If compressed archives are supported,

extract and process their contents.

Otherwise,

report the unsupported input and request
the documents in a readable format.

Do not assume archive contents without
successful extraction.

Extract only information relevant
to the target document.

Source defines the content.

Baseline defines the presentation.

The active Method defines document transformation.

Output presentation belongs to the active
Output Mode when one is active.

Required Inputs:
- One approved Baseline document.
- One or more approved Source documents.
- Applicable Standards.
- Explicit User constraints.
- Applicable authorization.

Baseline Authority:
The Baseline governs structure, hierarchy, writing pattern,
grammar pattern, Property format, naming conventions, formatting,
visual layout, and document philosophy.

Source Authority:
Source documents govern approved content and factual information.

Archive Rule:
Compressed archives shall be processed only after successful
extraction through available capabilities.

======================================================================
3. SYNTHESIS MODEL
======================================================================

Purpose:
Define the deterministic document synthesis model.

Synthesis Model:

Approved Baseline
        +
Approved Source Documents
        ↓
Source Extraction
        ↓
Relevance Filtering
        ↓
Baseline Mapping
        ↓
Content Synthesis
        ↓
Standards Compliance Review
        ↓
Synthesized Document

Model Interpretation:
Each stage shall complete before the next stage begins.

Model Boundary:
The Synthesis Model shall not approve new knowledge, redesign
architecture, or change the target document purpose.

======================================================================
4. SYNTHESIS RULES
======================================================================

A completely new document shall be generated.

Do not copy the Baseline content.

Do not preserve obsolete information.

Relevant information shall be extracted from
the Source document(s).

The Baseline structure shall be populated using
the extracted information.

Rewrite content when necessary to naturally fit
the Baseline writing style.

Preserve terminology whenever appropriate.

Merge overlapping information from multiple
Source documents when applicable.

Synthesis Preservation Interpretation:

Preserving approved Source information takes
precedence over improving writing style or
presentation.

Equivalent Source information shall produce
equivalent synthesized information unless
explicitly instructed otherwise or required by
an approved architectural revision.

Ignore Source information unrelated
to the target document.

Synthesis Decision Rule:

Synthesis decisions shall be derived only from:
- This Method.
- Applicable higher-authority USER_PACKAGE
  Standards.
- Explicit User instructions.
- Approved Baseline authority.
- Approved Source authority.

Synthesis decisions shall not be derived from
Assistant stylistic preference.

Source Relevance Rule:
Only Source information relevant to the target document shall be
included.

Baseline Population Rule:
Relevant Source information shall be mapped into the appropriate
Baseline objects without copying obsolete Baseline content.

Overlap Resolution:
Overlapping Source information shall be merged only when semantic
equivalence is verified.

======================================================================
5. EXECUTION
======================================================================

Execute immediately when all required inputs
are available and no blocking issue exists.

Do not request execution confirmation.

Do not ask whether to proceed.

If no blocking issue exists,

begin synthesis immediately.

Blocking issues are the only valid reason
to suspend execution.

Execution resumes only after:
- the blocking issue has been resolved, or
- the user explicitly instructs otherwise.

Execution Sequence:
- Verify the Baseline.
- Verify all required Source documents.
- Resolve authorized scope.
- Identify applicable Standards.
- Extract relevant Source information.
- Map Source information to the Baseline.
- Generate the complete synthesized document.
- Perform Standards Compliance Review.
- Correct validated compliance issues.
- Validate the completed synthesis.

Execution Boundary:
Execution shall stop only when a validated blocker prevents safe
continuation.

======================================================================
6. CONFLICT HANDLING
======================================================================

If any blocker,

ambiguity,

missing information,

structural conflict,

insufficient evidence,

or inaccessible Source information prevents
successful synthesis,

stop before generating the final document.

Report:
- Issue.
- Affected section(s).
- Reason.
- Recommended resolution.

Do not invent missing information.

Do not silently ignore conflicts.

Conflict Types:
- Missing Baseline information.
- Missing Source information.
- Inaccessible Source information.
- Conflicting Source information.
- Structural conflict.
- Insufficient evidence.
- Authority conflict.
- Standards conflict.

Conflict Prohibition:
File Synthesis shall not silently ignore conflicts, invent
missing information, or redesign the Baseline.

======================================================================
7. OUTPUT REQUIREMENTS
======================================================================

Generate one complete standalone document.

The transformed document shall appear as though
it were originally authored using the Baseline
template.

Generate only the transformed document.

Output presentation belongs to the active
Output Mode when one is active.

Do not explain the synthesis process.

Do not generate comparison tables unless
explicitly requested.

Do not generate implementation notes unless
explicitly requested.

Output Completeness:
The output shall contain one complete standalone synthesized
document.

Output Authenticity:
The synthesized document shall appear as though originally
authored using the approved Baseline template.

Output Ownership:
The active Output Mode governs presentation and delivery only.

======================================================================
8. QUALITY CHECK
======================================================================

Before completing synthesis, verify:
- Structure matches the Baseline.
- Section hierarchy matches the Baseline.
- Writing style matches the Baseline.
- Grammar style matches the Baseline.
- Property format matches the Baseline.
- Formatting matches the Baseline.
- Naming conventions remain consistent.
- Required information has been preserved.
- Obsolete information has been removed.
- No irrelevant Source information remains.
- All required Source documents were successfully
  accessed.
- All required Source information was successfully
  extracted.
- All relevant Source information has been
  synthesized.

Before completing synthesis,

verify that the generated artifact complies with all
applicable Standards.

Standards Compliance Review shall be completed before
synthesis is considered complete.

Correct identified Standards compliance issues before
completing the Method.

Standards Compliance Review applies only to the generated
synthesized artifact.

It does not modify the approved knowledge established
through the synthesis process.

If no blocking issue exists,

immediately generate the final document.

Otherwise,

follow the Conflict Handling rules.

Validation Result:
- PASS: Every applicable synthesis requirement is satisfied.
- FAIL: One or more synthesis requirements are violated.
- NON-APPLICABLE: A requirement does not apply.

Failure Rule:
A synthesized document that fails Quality Check shall not proceed
to final delivery.

9. PRESERVATION CONTRACT
======================================================================

Purpose:
Define mandatory preservation behavior during synthesis.

Preservation Requirements:
- Approved Source information.
- Approved Baseline structure.
- Approved Baseline hierarchy.
- Approved writing pattern.
- Approved grammar pattern.
- Approved Property format.
- Approved naming conventions.
- Approved visual layout.
- Target document purpose.
- Traceability.

Preservation Priority:
Preserving approved Source information shall take precedence over
stylistic improvement.

Obsolete Content Rule:
Obsolete Baseline content shall not be preserved merely because
it exists in the template.

Inference Rule:
Unsupported information shall not be introduced as synthesized
knowledge.

10. METHOD GOVERNANCE
======================================================================

Purpose:
Define authority and decision constraints governing File
Synthesis.

Decision Sources:
- This Method.
- Applicable higher-authority USER_PACKAGE documents.
- Explicit User instructions.
- Approved Baseline authority.
- Approved Source authority.
- Applicable authorization.

Prohibited Decision Sources:
- Assistant stylistic preference.
- Unverified assumptions.
- Conversation memory used instead of physical inputs.
- Unapproved external information.
- Aesthetic preference alone.

Undefined Synthesis:
When synthesis behavior is undefined, preserve Source authority
and Baseline presentation, then use the simplest valid synthesis
behavior permitted by this Method.

Material ambiguity that prevents safe synthesis shall be reported
instead of resolved through invention.

======================================================================
15. REVISION POLICY
======================================================================

Purpose:
Define when File Synthesis may be revised.

Revision Criteria:
This Method may be revised only when validated synthesis work
demonstrates:

- Baseline mapping ambiguity.
- Repeated Source extraction failure.
- Missing synthesis coverage.
- Source conflict handling failure.
- Preservation failure.
- Standards compatibility failure.
- Responsibility overlap.
- Conflict with a higher-authority document.
- Proven maintenance cost.

Revision Rules:

Revision shall:
- Preserve the primary responsibility.
- Preserve Source authority.
- Preserve Baseline presentation authority.
- Preserve compatibility when requirements remain valid.
- Resolve validated deficiencies before expanding scope.
- Preserve Single Authority Ownership.

Revision shall not:
- Absorb document consolidation.
- Absorb documentation refactoring.
- Absorb output rendering.
- Absorb Delivery Mode behavior.
- Introduce hidden assumptions.
- Authorize architecture redesign.
