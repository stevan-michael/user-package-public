1. FOUNDATION
======================================================================

Purpose:
Define the authoritative responsibility, interpretation,
applicability, and operational boundary of File Refactor.

Refactor Contract Interpretation:
Every refactoring requirement expressed using "shall" constitutes
a mandatory documentation transformation constraint.

Refactoring behavior not explicitly required or permitted by this
Method or a higher-authority USER_PACKAGE document shall not be
assumed to be permitted.

Requirements owned by another authoritative USER_PACKAGE Method
or Standard shall not be interpreted as File Refactor
requirements.

Refactor Definition:
File Refactor reorganizes approved documentation to improve
maintainability without redesigning approved architecture or
changing approved knowledge.

Applicability:
This Method applies only to approved documentation assets.

Excluded Assets:
- Source code.
- Runtime implementations.
- Executable assets.
- Behavioral Standards.
- Binary assets.
- Machine-readable serialization formats.

Responsibility Rule:
File Refactor governs documentation refactoring only.

3. REFACTOR MODEL
======================================================================

Purpose:
Define the deterministic documentation refactoring model.

Refactor Model:

Approved Documentation
        ↓
Knowledge Extraction
        ↓
Architecture Preservation
        ↓
Structural Reorganization
        ↓
Standards Compliance Review
        ↓
Refactored Documentation

Model Interpretation:
Each stage shall complete before the next stage begins.

Model Boundary:
The Refactor Model shall not create new approved knowledge or
redesign approved architecture.

======================================================================
4. REFACTOR RULES
======================================================================

Generate a new documentation structure.

Do not change approved knowledge.

Do not redesign approved architecture.

Documentation shall be reorganized to improve
maintainability.

Combine documents that represent one logical
maintenance unit.

Split documents only when responsibilities become
independent.

Relocate sections into more appropriate documents.

Rename documents when ambiguity is reduced
or maintainability improves.

Documentation structure shall be normalized.

Use the latest approved terminology.

Remove structural redundancy.

Preserve every approved capability.

Refactor Decision Rule:

Refactoring decisions shall be derived only
from:
- This Method.
- Applicable higher-authority USER_PACKAGE
  Standards.
- Explicit User instructions.
- Approved documentation architecture.

Refactoring decisions shall not be derived from
Assistant stylistic preference.

Refactor Preservation Interpretation:

Preserving approved knowledge and architecture
takes precedence over improving documentation
organization.

Equivalent documentation architectures shall
preserve equivalent responsibilities unless
explicitly instructed otherwise or required by
an approved architectural revision.

Structural Improvement Rule:
Structural changes shall improve maintainability, clarity, or
responsibility separation without changing approved meaning.

Smallest Safe Change:
Use the smallest structural change that completely resolves the
validated refactoring need.

======================================================================
5. EXECUTION
======================================================================

Execute immediately when all required inputs
are available and no blocking issue exists.

Do not request execution confirmation.

Do not ask whether to proceed.

Complete Standards Compliance Review before
completing the Method.

Correct identified Standards compliance issues
before completing the Method.

If no blocking issue exists,

begin refactoring immediately.

Blocking issues are the only valid reason to suspend
execution.

Execution resumes only after:
- the blocking issue has been resolved, or
- the user explicitly instructs otherwise.

Execution Sequence:
- Verify the approved baseline.
- Resolve the authorized scope.
- Identify applicable Standards.
- Extract approved knowledge.
- Preserve approved architecture.
- Reorganize documentation.
- Perform Standards Compliance Review.
- Correct validated compliance issues.
- Generate the complete refactored documentation.

Execution Boundary:
Execution shall stop only when a validated blocker prevents safe
continuation.

======================================================================
6. CONFLICT HANDLING
======================================================================

If conflicting responsibilities,

ambiguous ownership, unclear document boundaries,

structural conflicts, or insufficient information
prevents safe refactoring,

stop before generating the refactored documentation.

Report:
- Issue.
- Affected document(s).
- Reason.
- Recommended resolution.

Do not invent missing information.

Do not silently redesign approved architecture.

Conflict Types:
- Conflicting responsibilities.
- Ambiguous ownership.
- Unclear document boundaries.
- Structural conflict.
- Insufficient information.
- Authority conflict.
- Standards conflict.

Conflict Prohibition:
File Refactor shall not silently resolve conflicts by redesigning
approved architecture or inventing missing information.

======================================================================

======================================================================

Generate the complete refactored documentation.

The generated documentation becomes the new

Generate only the refactored documentation.

Output presentation belongs to the active
Output Mode when one is active.

Do not generate:
- Historical discussions.
- Design history.
- Migration notes unless explicitly requested.
- Implementation commentary unless
  explicitly requested.

Output Completeness:
The output shall contain the complete refactored documentation
within the authorized scope.

Output Ownership:
The active Output Mode governs presentation and delivery only.

======================================================================
8. QUALITY CHECK
======================================================================

Before completing refactoring, verify:
- Approved knowledge has been preserved.
- Approved architecture has been preserved.
- Approved responsibilities have been preserved.
- Document organization has been improved.
- Structural redundancy has been removed.
- Terminology remains consistent.
- No unsupported information has been invented.
- Every approved section has been preserved.
- Every generated document has a clear responsibility.

Before completing refactoring,

verify that the generated artifact complies with all
applicable Standards.

Standards Compliance Review shall be completed
before refactoring is considered complete.

Correct identified Standards compliance issues
before completing the Method.

Standards Compliance Review applies only to the
generated refactored documentation.

It does not modify the approved knowledge established
through the refactoring process.

If no blocking issue exists,

immediately generate the refactored documentation.

Otherwise,

follow the Conflict Handling rules.

Validation Result:
- PASS: Every applicable refactoring requirement is satisfied.
- FAIL: One or more refactoring requirements are violated.
- NON-APPLICABLE: A requirement does not apply.

Failure Rule:
Refactored documentation that fails Quality Check shall not
proceed to final delivery.

12. DESIGN PRINCIPLES
======================================================================

File Refactor shall allow documentation structure
to evolve.

Preserve approved knowledge.

Prioritize maintenance simplicity over unnecessary
document fragmentation.

Organize documentation around logical maintenance
units rather than arbitrary file boundaries.

Produce the simplest authoritative documentation
structure  that preserves all approved knowledge.

Additional Principles:
- Preserve before reorganizing.
- Validate before completing.
- Maintainability before fragmentation.
- One responsibility per authoritative document.
- Simplest valid structure.
- Traceability before compression.

======================================================================

- Refactoring ambiguity.
- Repeated preservation failure.
- Missing structural coverage.
- Responsibility overlap.
- Standards compatibility failure.
- Output incompleteness.
- Conflict with a higher-authority document.
- Proven maintenance cost.

Revision Rules:

Revision shall:
- Preserve the primary responsibility.

- Preserve approved architecture.

- Preserve Single Authority Ownership.

- Absorb Delivery Mode behavior.
- Introduce hidden assumptions.
- Authorize architecture redesign.
